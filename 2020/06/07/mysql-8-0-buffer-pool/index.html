MySQL 8.0 Buffer pool

Buffer Pool Introduction

CPU can access the memory with the buss directly.
CPU can not access or modify the data directly in the database file.
Before manipulating the data, the data must be loaded from the database file into the memory.
It is inefficient to reload the data from the database file every time you manipulate them.
So, caching the relative data in the memory will be a better choice.
Such a component denotes the buffer pool in the Innodb.
There are one or more independent buffer pool instances.
Every buffer pool instance consists of components following:
one or more buffer chunks: There are the frames(a block of memory) that can hold the data. Every frame has an independent control block in the chunk.
The free list: all unused frames are chained together here.
The lru list: all frames that hold the data (unmodified or modified) are chained together.
The flush list: all frames that hold the modified data are chained together. The frames in the flush list are also in the lru list too.
The page hash: The hash table speeds up looking frames up in the lru list.
double write buffer: Another block of memory independent of buffer chunks holds the frames that will be flushed into the file.
Mutexes: they protect the lists above in the concurrence environment.
Page Cleaner Thread: It flushes the dirty frames into double write buffer first. These frames will be flushed into dedicated space in the file in synchronous IO mode. Then, they will be written into the database file in asynchronous IO mode.
Read ahead: reading the pages ahead accelerates the efficiency.

Logical Buffer pool

Reading data

Read a page from the buffer pool.
Get the buffer pool instance from the page by a simple algorithm.
if the page is in the page_hash of the buffer pool instance, we get it. That means the page is in the lru list right now. The data is on the page.
If not, reading the page from the database file into the buffer pool is essential.
We need a free frame from the free list that will hold the page firstly.
Sometimes, without an unused frame fails the reading process. We will talk about it later.
After the preparation of the empty frame, file io will fill the frame with the data from the database file.

LRU List

if the length of the lru list is greater than 512, there is a young section with 5/8 of the length of the lru list at the end of it. The rest 3/8 is the old section.
if the length of the lru list is less than 512, there is just the old section.
the 1/4 at the head of the length of the young section is hot data.
the frame read from the file will be put into the old section first.
if we access the frame again, we move it from the old section to the head of the young section.
but if the frame is in the hot section of the young section, we do not move it to the head of the young section after we access it.
Only the frames at the rest of the young section will be moved to the head of the young section after they are accessed.

Get a free frame

Iteration 0:
if the free list has the frames, we get one frame.
if not, we will scan the end of the lru list and find an unused page. Then we retrieve it and return it into the free list. And we get the free frame again.
Also, it may fail again. we will flush a page into the file at the end of the lru list. We put it into the free list. Get the free frame again.
Again, it may fail again. Increase the iteration.
Iteration 1: it is almost the same as the iteration 0 except we scan the whole lru list. if the free frame is unavailable again, we increase the iteration and try it again.
Iteration > 1: it is almost the same as the iteration 1 except we sleep 10ms before flushing the data.
Iteration > 20: Print the error log for an unavailable free frame.

Evict frames

First, we try to free an uncompressed page of a compressed block from the unzip lru list. The compressed page is preserved, and it need not be clean. The algorithm for deciding which uncompressed page will be chosen is complicated. Then, we try to free a clean page from the common lru list.
if we do not get any free frames above, we need flush dirty frames. But, in user thread, we chose a single page from the end of the lru list every time. If the page chose is dirty, we flush it and put it into the free list again.

Page cleaner

The page cleaner thread flushes pages periodically background.
It puts replaceable pages at the tail of the lru list to the free list and flushes dirty pages at the tail of the lru list to the disk, then puts pages into the free list.
There multiple page cleaner threads. One is the coordinator thread that does management besides flushes pages and the others are worker threads.

Double Write Buffer

We flush dirty pages into the double write buffer first.
Then we flush these frames into the file in the system space with the synchronous io.
Now, there are complete frames in the file in the system space.
We flush these dirty pages in the double write buffer into the tablespace with asynchronous io.
After the asynchronous io finishes the flush job, we check the safety of the data and move the pages flushed into the free list.

File IO

The file io the basic component for the whole InnoDB also has been well designed.
There are two io mode, synchronous io, and asynchronous io.
The synchronous io is implemented with the pread/pwrite function and executed by the user threads always.
And the asynchronous io called AIO is implemented in two ways, the Simulated AIO and the Linux native AIO.
The Simulated AIO includes four the io-threads working in the background for the ibuf, the log, the reads, and the writing. The asynchronous requests are queued up in an independent array for four io-threads by the user thread. the user thread will return. Later the io-thread will process these requests with the synchronous io.
The Linux native AIO supported by the libaio library in the Linux also has a helper thread working on the four arrays above in the background. The user thread queues the AIO requests and submits them to the system in the array. The helper thread only collects the completed IO requests and calls the completion routine on it.
