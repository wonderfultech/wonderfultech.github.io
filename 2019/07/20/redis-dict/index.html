Redis Dict 字典

Posted on 2019年7月20日2019年8月30日by jasonpeng

leetcode 数据库题目全部题解

The internals of the Dict is HashTable. HashTable holds data in dict.

The features of HashTable:

linkedlist resolving conficts;

the size of hashtable is 2^N;

can hold multiple types of data;

Actually, dict holds two hashtables inside it at most. Usually, there is only one hashtable. But, when dict is doing the rehash, it holds two hashtables.

There are three hash functions used in dict. We will discuss them in later posts.

Thomas Wang's 32 bit Mix Function: Integer mix function

/* Thomas Wang's 32 bit Mix Function */
unsigned int dictIntHashFunction(unsigned int key)
{
key += ~(key << 15);
key ^= (key >> 10);
key += (key << 3);
key ^= (key >> 6);
key += ~(key << 11);
key ^= (key >> 16);
return key;
}

MurmurHash2: long string hash function. I never find any materials talking about the principle behind the MurmurHash. We can get the code, test cases. But we can not understand how it works, it has a reverse function or not.

unsigned int dictGenHashFunction(const void *key, int len) {
	/* 'm' and 'r' are mixing constants generated offline.
	They're not really 'magic', they just happen to work well. */
	uint32_t seed = dict_hash_function_seed（默认是5381）;
	const uint32_t m = 0x5bd1e995;
	const int r = 24;
	/* Initialize the hash to a 'random' value */
	uint32_t h = seed ^ len;
	/* Mix 4 bytes at a time into the hash */
	const unsigned char *data = (const unsigned char *)key;
	while(len >= 4) {
		uint32_t k = *(uint32_t*)data;
		k *= m;
		k ^= k >> r;
		k *= m;
		h *= m;
		h ^= k;
		data += 4;
		len -= 4;
	}
	/* Handle the last few bytes of the input array */
	switch(len) {
		case 3: h ^= data[2] << 16;
		case 2: h ^= data[1] << 8;
		case 1: h ^= data[0]; h *= m;
	};
	/* Do a few final mixes of the hash to ensure the last few
	* bytes are well-incorporated. */
	h ^= h >> 13;
	h *= m;
	h ^= h >> 15;
	return (unsigned int)h;
}

Bernstein hash djb2: short string hash function. It is a very concise code, but also I find none materials about its principle. I think the complicated parts of djb2 and murmurhash are the "while loop " portion. Both of them can consume variable length string.

unsigned int dictGenCaseHashFunction(const unsigned char *buf, int len) {
unsigned int hash = (unsigned int)dict_hash_function_seed（默认是5381）;
while (len--)
//hash*33+c
hash = ((hash << 5) + hash) + (tolower(*buf++)); /* hash * 33 + c */
return hash;
}

An important criterion for hashtable to expand itself is the load factor of it. when the load factor is bigger than the threshold, hashtable begin to expand itself. After the expansion is done, load factor regresses into a low level.

The criterion for rehashing. we cite the code here.

/* If we reached the 1:1 ratio, and we are allowed to resize the hash
* table (global setting) or we should avoid it but the ratio between
* elements/buckets is over the "safe" threshold, we resize doubling
* the number of buckets. */
if (d->ht[0].used >= d->ht[0].size &&
(dict_can_resize ||
d->ht[0].used/d->ht[0].size > dict_force_resize_ratio)
)
{
// 新哈希表的大小至少是目前已使用节点数的两倍
// T = O(N)
return dictExpand(d, d->ht[0].used*2);
}

the load factor of hashtable 0 is bigger than and equal to 1

and

( dict_can_resize or the load factor of hashtable 0 is bigger than and equal to 5 )

The new size of expanded hashtable.

The new size is more than double size of original size at. Then the load factor is less than 0.5.

Now, dict will finish the rehash in many iterations. In every iteration, it will transfer nodes in only one bucket in hashtable[0] to another bucket in hashtable[1]. We need emphasize that the rehash will last a period of time. When some related functions are executed, a single step of rehash will be executed. Whole progress of the rehash will not be accomplished instantly when it is triggered.

the routine of rehash in single step like this:

rehashidx indicates which bucket need to transfered in hashtable[0].
for very node de in bucket[rehashidx]:
    get hashcode of de->key;
    get position of de in hashtable[1]:h=hashcode & ht[1].sizemask;
    put de node into the bucket[h] of hashtable[1];
    size of hashtable[0] and hashtable[1] is updated by one.
reset bucket[rehashidx] of hashtable[0] into empty.
rehashidx++;

Now, we know the criterion and routine of rehash. But there are two pending questions: when does it trigger the criterion? where the rehash is finished?

The function _dictKeyIndex situates the bucket for a key. It will trigger the criterion.

There are four fuctions will do the routine of rehash.

dictAddRow: put a new key into dict.

dictGenericDelete: delete a key.

dictFind : find a key.

dictGetRandomKey: return a random node in the dict.
